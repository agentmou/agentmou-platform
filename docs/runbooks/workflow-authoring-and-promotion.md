# Workflow Authoring and Promotion

Use this runbook when creating or updating an n8n-backed workflow template that
should become a real Agentmou asset.

## Outcome

At the end of this procedure you should have:

- one canonical workflow template committed to Git
- one `manifest.yaml` that declares runtime and catalog metadata
- one sanitized `workflow.json`
- optional pack linkage
- at least one disposable-tenant validation run

## Operating Rules

- Git is the canonical source of workflow definitions.
- Do not commit one workflow JSON per tenant.
- The shared n8n instance is runtime infrastructure, not the control plane.
- Default to `platform_managed` credentials unless the workflow requires a
  documented `n8n_native_exception`.

## Inputs

You need:

- a target workflow ID
- the intended runtime owner and credential strategy
- the list of required connectors
- at least one disposable validation tenant

## Procedure

### 1. Prototype

Prototype the workflow in whichever environment is fastest:

- n8n editor
- n8n MCP tooling
- a local scratch export

Do not treat the prototype as canonical yet.

### 2. Export

Export the workflow JSON from n8n after the flow shape is stable.

Keep the exported file local until you sanitize it.

### 3. Sanitize

Before committing `workflow.json`, remove or normalize values that should not
be treated as source-of-truth state.

Check for:

- top-level IDs generated by n8n
- `active` flags
- created/updated timestamps
- version IDs or archive fields
- tenant-specific names
- real credentials or credential values
- environment-specific URLs that should be parameterized

Credential references are allowed only as stubs. Never commit tenant secrets.

### 4. Place the Asset

Create or update:

- `workflows/public/<workflow-id>/workflow.json`
- `workflows/public/<workflow-id>/manifest.yaml`
- `workflows/public/<workflow-id>/README.md`

Use `workflows/planned/` only for non-installable planning assets.

### 5. Declare Runtime Metadata

In `manifest.yaml`, set the runtime section explicitly:

- `requiredConnectors`
- `credentialStrategy`
- `installStrategy`
- `runtimeOwner`

For the current platform model:

- use `runtimeOwner: n8n` for deterministic n8n workflows
- use `installStrategy: shared_n8n_per_installation` for tenant installs

If the workflow needs n8n-native credentials, document why in the README and
set `credentialStrategy: n8n_native_exception`.

### 6. Declare Catalog Metadata

Populate the nested `catalog` block so the API can map the operational manifest
to the shared marketplace contract without leaking raw manifest details.

At minimum, define:

- summary
- integrations
- output
- use case
- risk level
- changelog
- nodes overview

### 7. Link Packs

If the workflow belongs in a pack, update the relevant file under
`catalog/packs/`.

Do not create tenant-specific pack variants in Git.

### 8. Validate Locally

Run the relevant checks for contracts and manifest loading:

```bash
pnpm --filter @agentmou/contracts test
pnpm --filter @agentmou/catalog-sdk test
pnpm --filter @agentmou/api test
```

Add or update tests if the asset introduces new mapping or schema behavior.

### 9. Install Into a Disposable Tenant

Validate the promotion path end to end:

1. install the workflow through the platform
2. confirm one `workflow_installations` row is created
3. confirm a concrete `n8nWorkflowId` is stored
4. trigger at least one run
5. inspect run status and execution logs

If you validate with multiple tenants, the same `templateId` should result in
different installation rows and different `n8nWorkflowId` values.

### 10. Validate Cleanup

Uninstall the workflow or use the guarded validation-fixture cleanup path.

Confirm:

- the remote n8n workflow is removed or 404s idempotently
- repeatable schedules are removed if present
- local installation rows are deleted only after external cleanup succeeds

## Promotion Checklist

- `workflow.json` is sanitized
- `manifest.yaml` includes explicit runtime metadata
- `README.md` documents credential strategy and runtime ownership
- no tenant-specific values were committed
- tests cover schema/mapping behavior
- install and cleanup were exercised on a disposable tenant

## Notes

- Treat Git as canonical even if the first draft was created in n8n.
- Prefer documenting exceptions over hiding them. A workflow that still needs
  n8n-native credentials is acceptable if the exception is explicit.
