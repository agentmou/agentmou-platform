# Catalog System: Marketplace & Templates

## Overview

The catalog system manages the **discovery, definition, and installation** of agents, workflows, and packs. It implements a **three-layer architecture** separating operational truth (source), demo/preview (snapshot), and marketing presentation.

**Key concepts**:
- **Operational catalog**: YAML manifests in `catalog/` (source of truth)
- **Demo catalog**: Generated JSON snapshot used by web app (read-only)
- **Availability tiers**: planned, preview, available (control feature rollout)
- **Categories**: 8 business domains (sales, support, marketing, etc.)
- **Templates**: Reusable starting points for agents + workflows

---

**Location**: `catalog/` directory (Git-tracked)

**Structure**:
```

├── agents/
│   ├── inbox-triage/
│   │   ├── manifest.yaml          (metadata + capabilities)
│   │   ├── prompt.md              (system prompt for agent)
│   │   ├── policy.yaml            (governance rules)
│   │   └── README.md              (user-facing docs)
│   ├── sales-pipeline-analyzer/
│   │   ├── manifest.yaml
│   │   ├── prompt.md
│   │   ├── policy.yaml
│   │   └── README.md
│   └── [more agents...]
│
├── packs/
│   ├── support-bundle/
│   │   ├── manifest.yaml
│   │   ├── includes.yaml          (list of agents/workflows)
│   │   └── README.md
│   └── [more packs...]
│

```

**Why YAML?** Version control friendly; easy to review in pull requests; human-readable

### Layer 2: Demo Catalog (Generated Snapshot)

**Location**: `apps/web/lib/demo-catalog/operational-ids.gen.json`

**Purpose**: Pre-computed, optimized JSON used by web frontend for:
- Fast marketplace browsing (no backend queries)
- SEO optimization (static metadata)
- Offline support (bundled in frontend build)

**Generated By**: `pnpm demo-catalog:generate`
```bash
# Reads all catalog/*.yaml files
# Generates operational-ids.gen.json with:
# - All agent/workflow metadata
# - Resolved dependencies
# - Category mappings
# - Availability status

# Checked in CI:

```

**Example Generated JSON**:
```json
{
  "agents": [
    {
      "id": "inbox-triage",

      "version": "0.1.0",
      "description": "Automatically categorize, prioritize, and label incoming Gmail messages for support teams",
      "category": "support",
      "availability": "available",
```

## Manifest Structure (manifest.yaml)

### Example: Inbox Triage Agent

```yaml
# Unique identifier (lowercase, hyphenated)
id: inbox-triage

# Display name
name: Inbox Triage

# Version (semver)
version: 0.1.0

# One-line description
description: Automatically categorize, prioritize, and label incoming Gmail messages for support teams

# Business domain
category: support

# Author/maintainer
author: Agentmou Team

# Keywords for search
tags:
  - email
  - gmail
  - automation
  - triage

# What connectors/APIs can this agent use?
capabilities:
  - read: gmail                # Can read emails
  - write: gmail               # Can apply labels, archive
  - labels: true              # Can create/manage labels

# How is this agent triggered?
triggers:
  - type: schedule
    cron: '*/15 * * * *'      # Every 15 minutes

  - type: webhook
    event: gmail.new_message   # Or manual trigger

# Runtime configuration
runtime:
  requiredConnectors:
    - gmail                     # Agent needs Gmail connected
  credentialStrategy: platform_managed  # We manage OAuth tokens
  installStrategy: platform_managed_installation
  runtimeOwner: agent_engine    # Executed in our agent engine
  linkedWorkflows:
    - wf-01-auto-label-gmail   # Optional: call n8n workflow

# Catalog & marketplace metadata
catalog:
  outcome: Keep shared inboxes organized without manual triage
  domain: support

  # Input/output for documentation
  inputs:
    - unread_emails
    - connector_context
  outputs:
    - classifications
    - label_actions
    - response_drafts

  riskLevel: low                # Policy implications

  # HITL required for email send?
  hitl: optional

  # Business metrics
  kpis:
    - name: Time to first triage
      description: Minutes until a new message is categorized
    - name: Label coverage
      description: Percentage of unread messages labeled automatically

  # Complexity: XS, S, M, L, XL
  complexity: S

  # Release channel
  channel: stable               # stable | beta | experimental

  # Setup time (minutes)
  setupTimeMinutes: 15

  # Pricing (null = included in base plan)
  monthlyPrice: null

  # Availability tier
  availability: available       # planned | preview | available

  # Target audience
  audience: business

  # Where is this defined?
  source: manifest              # manifest | generated | external

  # UI grouping
  catalogGroup: support
  family: inbox                 # Inbox agents family
  subdomain: email-operations

  # Search tags
  tags:
    - email
    - gmail
    - automation
    - triage

  # Show on landing page?
  featured: true

  # Who can see this?
  visibility: public            # public | private | enterprise

# Metadata
metadata:
  createdAt: '2026-03-07'
  updatedAt: '2026-03-07'
```

## Prompt File (prompt.md)

System prompt loaded at runtime for the agent.

**Location**: `catalog/agents/{id}/prompt.md`

**Variables**: Tenant-specific substitutions with `{{variable}}` syntax

**Example** (inbox-triage/prompt.md):
```markdown
You are an email triage assistant for {{org_name}}.

Your primary goal is to help keep shared inboxes organized by automatically
categorizing and prioritizing incoming messages.

## Your responsibilities:
1. Read unread emails from the inbox
2. For each email, determine:
   - **Priority**: high, medium, or low
   - **Category**: support, sales, general, spam
   - **Action**: reply, delegate, archive, or flag
   - **Suggested labels**: from {{enabled_labels}}

## Rules:
- **Urgent customer issues** (e.g., "account compromised", "immediate help needed") → High priority
- **Spam indicators**: unsubscribe links, promotional offers, no personalization → Spam category
- **Unknown priority** → Default to Medium
- **Multi-part emails**: Focus on first part for classification

## Output format:
Return a JSON array:
\`\`\`json
[
  {
    "messageId": "...",
    "priority": "high|medium|low",
    "category": "support|sales|general|spam",
    "action": "reply|delegate|archive|flag",
    "labels": ["urgent", "followup"],
    "confidence": 0.95,
    "reasoning": "..."
  }
]
\`\`\`

## Constraints:
- Maximum 100 emails per run (performance)
- Only apply labels from the {{enabled_labels}} list
- Do NOT send emails or delete messages
```

### Variable Substitution

**Tenant Config** (stored in installation.config):
```json
{
  "org_name": "Acme Corp",
  "enabled_labels": ["urgent", "followup", "archived"],
  "max_emails": 50
}
```

**Substituted Prompt** (passed to Planner):
```markdown
You are an email triage assistant for Acme Corp.
...
   - **Suggested labels**: from urgent, followup, archived
...
- Maximum 50 emails per run
```

## Policy File (policy.yaml)

Governance rules enforced during execution.

**Location**: `catalog/agents/{id}/policy.yaml`

**Example** (inbox-triage/policy.yaml):
```yaml
policies:
  # Reading emails is always allowed
  gmail.read:
    allowed: true
    riskLevel: 'low'
    requiresApproval: false
    quota:
      daily: 10000
      perRun: 500

  # Labeling emails is allowed but medium risk
  gmail.write:
    allowed: true
    riskLevel: 'medium'
    requiresApproval: false
    # (could be true if tenant wants approval for every label)

  # Sending emails requires approval
  gmail.send:
    allowed: true
    riskLevel: 'high'
    requiresApproval: true
    quota:
      perRun: 10

  # Deleting is not allowed
  gmail.delete:
    allowed: false
    reason: 'Data safety: no permanent deletions'

  # Invoking external APIs (e.g., Slack notification)
  external_api.invoke:
    allowed: true
    riskLevel: 'medium'
    requiresApproval: false
    cost_threshold: 1.0  # Require approval if > $1

constraints:
  # Safety cap on run cost
  - name: 'max_cost_per_run'
    value: 50.0  # USD
    enforcement: 'block'
    description: 'Prevent runaway costs'

  # Max API calls per tenant per day
  - name: 'max_api_calls_per_day'
    value: 1000
    enforcement: 'warn'
    description: 'Alert if quota approaching'
```

## Availability Tiers

Control feature rollout and visibility.

| Tier | Visibility | Installable | Use Case |
|------|------------|------------|----------|
| **planned** | Admins only | No | Announced but not ready (coming Q2) |
| **preview** | Opted-in users | Yes | Beta testing, limited customers |
| **available** | All users | Yes | General availability, production-ready |

**Example**:
- Inbox Triage v0.1.0 → planned (announce on roadmap)
- Inbox Triage v0.2.0 → preview (send to beta users for feedback)
- Inbox Triage v0.3.0 → available (full release)

**Configuration** (manifest.yaml):
```yaml
catalog:
  availability: available  # ← Change this to roll out
```

## Creation & Promotion Flow

### Creating a New Agent

```
1. Scaffold from template
   $ cp -r templates/product-agent-simple catalog/agents/my-new-agent

2. Edit manifest.yaml
   - Set id, name, description
   - Define capabilities, triggers
   - Set availability: planned (initial)

3. Write prompt.md
   - Define agent behavior
   - Include {{variables}} for tenant config

4. Define policy.yaml
   - Set allowed actions + risk levels
   - Add constraints if needed

5. Create README.md
   - User-facing documentation
   - Setup guide, examples

6. Test locally
   pnpm dev  (spin up full stack)
   - Manually run agent
   - Verify prompt, policies

7. Commit and push
   git commit -m "feat: add my-new-agent (planned)"

8. Check demo catalog sync
   pnpm demo-catalog:check  (should pass)
```

### Promotion: planned → preview → available

**From planned to preview**:
1. Update manifest.yaml: `availability: preview`
2. Invite beta users to test
3. Gather feedback in Linear/GitHub issues
4. Fix issues, iterate

**From preview to available**:
1. Update manifest.yaml: `availability: available`
2. Announce on blog/changelog
3. Add to featured section (if applicable)
4. Monitor adoption + success metrics

**Each promotion = new version**:
```yaml
version: 0.1.0  # planned
version: 0.2.0  # preview
version: 0.3.0  # available (stable)
```

## operational-ids.gen.json: Demo Catalog

Auto-generated file that serves the web app's catalog endpoints.

**Generated by**: `scripts/generate-operational-catalog-ids.ts`

**Process**:
1. Scan `catalog/agents/**/*.yaml`
2. Load each manifest + metadata
3. Resolve dependencies (linkedWorkflows)
4. Generate JSON with normalized structure
5. Write to `apps/web/lib/demo-catalog/operational-ids.gen.json`

**CI Check** (`pnpm demo-catalog:check`):
```bash
# 1. Regenerate
pnpm demo-catalog:generate

# 2. Compare with committed version
git diff --exit-code -- apps/web/lib/demo-catalog/operational-ids.gen.json

# 3. Fail if mismatch (someone edited YAML without regenerating)
```

**Why?** Ensures web frontend always has fresh catalog without requiring API call

## API Endpoints (Control Plane)

### GET /api/catalog/agents

Fetch all agents (uses demo-catalog)

**Query Parameters**:
- `category`: Filter by category (e.g., "support")
- `availability`: Filter by tier (e.g., "available")
- `featured`: Show only featured agents

## Search & Discovery

### Marketplace Search (Web UI)

**UX Flow**:
1. User types "email" in search bar
2. Frontend searches `operational-ids.gen.json` in-memory
3. Matches against name, description, tags
4. Returns matching agents + workflows
5. User clicks to view details or install

**Why client-side search?**
- Fast (no network call)
- Works offline
- No backend load
- Works even if API down

### Category Browsing

**UX Flow**:
1. User clicks "Support" category
2. Frontend filters `operational-ids.gen.json` by `category: 'support'`
3. Shows all support agents + workflows
4. User can further filter by `availability`

## Complexity Levels

Rating system for user guidance on setup difficulty.

| Level | Setup Time | Expertise | Examples |
|-------|-----------|-----------|----------|
| **XS** | <5 min | None | Pre-built, no config |
| **S** | 5-15 min | Basic | Select connectors, basic settings |
| **M** | 15-60 min | Intermediate | Custom prompts, policy editing |
| **L** | 1-4 hours | Advanced | Workflow integration, API setup |
| **XL** | 4+ hours | Expert | Custom tool dev, complex policies |

**Used for**:
- Marketplace filtering ("Show me simple agents")
- Onboarding guidance
- Documentation level
